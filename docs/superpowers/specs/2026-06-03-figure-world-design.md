# 公仔世界（Figure World）Design Spec

> 狀態：草稿待 review · 日期 2026-06-03
> 平行於：旅程世界（Slice P，已上線，location 驅動）、居家世界（`docs/home-world-design`，草稿，completion-count 驅動）
> 視覺脈絡：`docs/superpowers/notes/2026-05-28-gamification-exploration.md`（珍惜驅動、零懲罰、工具/收集分離哲學）
> 視覺 brief（本 spec 一起 push 的 4 張 mockup）：`docs/superpowers/notes/assets/2026-06-03-figure-world-*.html`

## 0. 一句話

把使用者「完成了多少習慣、哪些面向」渲染成一隻**會被你養大、越來越可愛的萌寵**——9 大 GENESIS+IO 健康面向各自餵養萌寵的不同部位 / 裝飾，完成習慣＝餵養，萌寵隨之長大、進化、長出反映你習慣組成的外型。**零 location、零拍照、零隱私顧慮，所有使用者開箱即玩**（與居家世界同屬「人人能玩」陣營，與 location-gated 的旅程世界互補）。

## 1. 三個世界的關係（共用契約 — 與居家世界完全一致）

公仔世界是三個可切換遊戲化「世界」之一（旅程 / 公仔 / 居家），**一次只玩一種**。World-switch 的底層契約**已由居家世界 spec §1 定案**，本 spec **完全沿用、不另立**，只摘要對齊點：

```diff
model User {
+ activeWorld String?   // null = 還沒選世界（首次迎賓）。選後為 'home' | 'figure' | 'journey'
}
model TaskHistory {
+ world String?   // 完成當下 activeWorld 快照；null = 此機制上線前 / 選世界前
}
```

- **共同序章（Prologue）**：選世界前的完成（`world IS NULL`）＝所有世界共享的起步基礎。公仔世界的某 domain 完成數 ＝ `count(world IS NULL AND domain=X) + count(world='figure' AND domain=X)`。與居家世界同公式、只是 `world='figure'`。
- **World Picker** ＝ 跨 session 共用元件（首次迎賓並排「居家/公仔/旅程」+ 之後隨時切換、零確認）。公仔世界只需提供選單資料介面：`key='figure'`、名稱、一句話、預覽縮圖、當前規模摘要（如「Lv.12 小芽」）。
- 切換不回溯、舊世界不歸零、切回繼續。

> ⚠️ **F1（world-switch 基礎）與居家世界 H1 是同一塊東西**。兩個 session 不可各做一套。協調方案見 §9.2：建議抽成獨立的「world-switch 基礎」PR，居家 + 公仔都依賴它。本 spec 把契約釘成對齊基準（與居家 spec §1 逐字一致）。

## 2. 範圍

### 做（公仔世界 v1，分階段見 §8）
- 側邊欄新增 `figure`（公仔 / 夥伴）view（與 daily/journey/home/stats 同層 `setCurrentView`）。
- **一隻萌寵**（非 9 隻）：9 大 domain = 同一隻萌寵的 9 個面向 / 部位 / 裝飾（§4）。
- **三軸成長**（§4）：① 成長階段（總完成數）② 領域部位（各 domain 完成數）③ 形態傾向（domain 組成）。
- **6 階成長弧**：蛋 → 萌芽 → 幼生 → 成長 → 成熟 → 綻放（之後無上限，§5）。
- **餵養回饋**：完成習慣 → 食物圖示飛向萌寵 + 愛心 + 安靜的成長進度（§3、§7）。
- **多萌寵物種**：小芽（嫩芽系）/ 奶茶（貓系）/ 啵啵（史萊姆系）… 可選 / 可切換的**外皮**，共用同一成長引擎（§6）。
- World 切換器（§1，共用）。
- `figureWorld` 純函式衍生 lib（階段判定、部位解鎖、形態傾向、進度）+ TDD。
- 萌寵 SVG 美術（可組合三層：基底 × 階段 × 配件）——**全 SVG、零插畫師**（§7）。
- 空狀態（蛋還沒孵 → 引導完成第一個習慣）。

### 不做（後續 / 其他 session）
- ❌ **9 隻萌寵（每 domain 一隻）** — 明確否決：拆 9 隻會稀釋情感、讓人忽略弱項 domain（§4.1）。
- ❌ 公仔站進居家房間裡走動 ＝ 與居家世界整合時談（居家 spec §9.3 已留「角色層」擴充餘地）。
- ❌ 萌寵對戰 / 社交 PK / 拜訪別人的萌寵 ＝ 未來社交 slice。
- ❌ 課金轉蛋 / 付費裝飾 ＝ 不做（珍惜驅動 ≠ 課金）。
- ❌ 任何懲罰（沒餵萌寵會餓死 / 生病 / 跑掉）＝ **絕不做**。零懲罰（§3）。
- ❌ 走路 / 步數 ＝ 已從整個方向移除。
- ❌ AI 生成美術 ＝ 不做（同一角色跨 6 階段一致性 AI 現難穩定；改用可組合 SVG）。

## 3. 設計哲學（承接 gamification note）

1. **普惠優先** — 與居家世界同：只靠完成數，**人人開箱即玩**（旅程世界要 opt-in location，門檻高）。
2. **最強情感槓桿** — 養一隻會呼吸眨眼的小生物 ＝ 養育本能（Pikmin / Finch 的核心），比城市 / 房子更黏、最適合「年輕化、遊戲化」訴求。
3. **養成感 > 數字** — 萌寵親手養大、外型由你的習慣決定，ownership 強。
4. **公仔 = 習慣的肖像** — 同一顆蛋，因習慣組成不同長成不同樣子（§4.3）。呼應 Slice E 身分宣告：習慣是身分的證據，萌寵把它具象化。
5. **工具/收集分離** — 「今天」分頁純清單；「公仔」分頁才是遊戲化。完成卡標注「餵養 X 領域」把工具動作接到養成。
6. **零懲罰** — 沒餵不會死、不會餓壞、不會跑掉，只是不長、表情中性。永遠用「想看牠長大」拉回，不用「怕牠死」威脅。

## 4. 萌寵成長：三軸 + domain 映射

### 4.1 一隻萌寵，不是 9 隻【定案】

9 大 domain ＝ **同一隻萌寵的 9 個面向**（部位 / 裝飾），不是 9 隻。理由：
- 情感集中（寵物的魔力來自「一隻你愛的」，拆 9 隻稀釋 9 倍）。
- **偏食的萌寵長得歪 ＝ 功能不是 bug**：只做飲食 → 只有嫩芽、沒肌肉沒睡帽 → 視覺上逼你補齊弱項，正是 cofit 全面健康的訊息。9 隻會讓人「只養一隻、其他擺爛」。
- 美術成本低（一隻基底 + 配件層，不是 9 套）。

### 4.2 三軸變化

| 軸 | 由什麼驅動 | 視覺 |
|---|---|---|
| **① 成長階段** | 公仔世界**總**完成數（序章＋figure） | 蛋→萌芽→幼生→成長→成熟→綻放（變大、變成熟） |
| **② 領域部位** | **各 domain** 在公仔世界的完成數 | 各 domain 長各自的裝飾（見下表） |
| **③ 形態傾向** | domain **組成 / 主導** | 里程碑進化成不同樣子（均衡型 / 運動型 / 靜養型…） |

### 4.3 domain → 萌寵部位/裝飾映射

| Domain | 部位 / 裝飾意象 |
|---|---|
| 飲食 | 頭頂嫩芽更茂盛 / 圓肚 / 健康光澤 |
| 運動 | 壯的四肢 / 啞鈴 / 頭帶 / 腹肌 |
| 壓力與睡眠 | 睡帽 / 慵懶瞇眼 / 月亮motif |
| 認知與智慧 | 眼鏡 / 書本 / 大一點的頭 |
| 社交互動 | 身邊有小夥伴 / 暖色 |
| 心靈 | 蓮花 / 柔光環 / 禪定姿 |
| 環境 | 身邊飄葉 / 小盆栽 |
| 基因與腸道 | 飽滿圓潤 / 元氣腮紅 |
| 職涯與平衡 | 小領帶 / 平衡感配件 |

- 對不到 domain 的完成（`categoryToDomain → 'other'`）＝ 算進總完成數（推進成長階段），但不長特定部位；可累積到「通用裝飾池」（帽子 / 小物，任意搭配）。
- 重用既有 `categoryToDomain()`（Slice P 已寫，含髒值正規化）——不重造。

### 4.4 形態傾向（第三軸細節）

- 同一顆蛋，因 domain 組成不同，里程碑時長成不同**傾向**：均衡型（各部位齊全、圓滿）/ 運動型（壯碩、啞鈴）/ 靜養型（睡帽、蓮花、慵懶）…
- v1 可先只做「部位疊加」（軸②），**形態傾向（軸③）列為 F4 / v2**，因為它需要更多美術變體。先有「部位反映習慣」就已經是「習慣肖像」的 80%。
- 形態由「主導 domain」決定 or「平衡度」決定 → **開放問題（§9.3）**。

## 5. 成長階段與門檻

### 5.1 6 階弧（純函式 `figureWorld.js`）

```js
// 集中常數，spec 給初值、日後微調（對齊 journeyWorld.js / homeWorld.js 風格）
const STAGE_THRESHOLDS = [0, 3, 10, 25, 50, 90];  // 總完成數 → 階段 0..5
// 階段名：蛋 / 萌芽 / 幼生 / 成長 / 成熟 / 綻放
const PART_UNLOCK_THRESHOLDS = [1, 6, 18, 45];     // 各 domain 完成數 → 該部位 1/2/3/4 級
```

- `stageOf(totalCount)` → 0..5；封頂（>90）後持續完成 → 解鎖季節限定型態 / 稀有裝飾（不封頂，款式輪替，類似 journeyWorld `buildingStyleIndex`）。
- `partLevel(domainCount)` → 該 domain 部位的發展程度（0..4）。
- **解鎖即恭喜、零懲罰**：長出的部位不會因為之後沒做而消失。

### 5.2 階段定義（視覺見 growth-stages mockup）

| 階段 | 門檻（總數） | 樣子 |
|---|---|---|
| 蛋 | 0–2 | 米色蛋 + 頂端冒一點芽 |
| 萌芽 | 3–9 | 破殼、圓身體、一片葉、大眼 |
| 幼生 | 10–24 | 離殼、雙葉、短手臂、會笑（＝主視覺常見態） |
| 成長 | 25–49 | 抽高、長腳、花苞、部位明顯 |
| 成熟 | 50–89 | 茂密嫩芽＋綻放花、飄葉 |
| 綻放 | 90+ | 花冠＋柔光環＋花瓣，無上限續長 |

## 6. 多萌寵（物種）模型

### 6.1 物種 = 外皮，共用成長引擎

- 小芽（嫩芽系）/ 奶茶（貓系）/ 啵啵（史萊姆系）… 是**不同基底 SVG**，但跑**同一個 figureWorld 引擎**（同階段門檻、同部位/配件系統）。
- 視覺見 `multi-pet` mockup：三層獨立——**基底（物種）× 階段層 × 配件層**——疊出所有組合，加新萌寵只多畫一個基底。

### 6.2 切換物種：re-skin 同一份成長（v1 建議）

- 公仔世界只有**一個成長池**（序章＋figure 完成數）。「目前養的萌寵」是渲染這份成長的**物種皮膚**。
- 切換物種 ＝ 把同一份成長**換一個物種重新表現**（Lv.12 小芽 → Lv.12 奶茶），**不歸零、零損失**——符合零懲罰。
- 新物種用「里程碑 / 連續天數」解鎖（給長期動機），不課金。
- **替代方案**（每隻獨立成長池）＝ 較像收集但會分散、且要 per-pet schema → 列**開放問題（§9.4）**，v1 傾向「re-skin 同池」。

### 6.3 Schema（公仔世界專屬，最小化）

```diff
model User {
+ activeFigure String?   // 'sprout' | 'cat' | 'slime' …；null = 預設 sprout
}
```

- **成長與部位解鎖純從完成數衍生**（§5），零 schema、與旅程/居家世界一致、無資料遷移。
- 只持久化「目前養哪隻」（無法從完成數推導）。
- 物種**解鎖狀態**也從完成數 / 連續天數衍生（不存）。
- **使用者手動佈置的裝飾**（若 F3 做「自由搭配裝飾」）→ 才需要一張 `FigureDecorPlacement`（mirror 居家 `HomeFurniturePlacement`）。v1 部位/配件都**自動由 domain 衍生**，先不做手動佈置 → F3 再評估。

## 7. 視覺系統（零插畫師）

- **可組合 SVG 三層**：基底身體（物種）+ 階段變形 + domain 配件層。不畫每種組合、用程式疊。
- 美術語言（已在 mockup 確立）：徑向漸層做立體、大眼＋雙高光、腮紅、柔和投影、無生硬黑邊；idle 動畫（呼吸 / 眨眼 / 嫩芽搖擺 / 史萊姆 wobble）。
- 配件用 SVG path + domain 代表色（重用 `CATEGORY_CONFIG` 色票；**lib 色票 class 已在 tailwind content glob 內**，見 `fix/tailwind-scan-lib`）。
- 餵養瞬間：完成 → 食物圖示 feed-pop 飛向萌寵 + 愛心浮起（強但短，~1s，之後安靜）。
- **這版 SVG 是 v1 正式美術，也是未來插畫師的 brief**：有預算時把「三層系統 + 6 階弧 + 9 配件」交給插畫師重畫精緻版，架構不變。

## 8. 分階段實作（對齊 O→P→Q / 居家 H1→H4）

| 階段 | 範圍 | schema | 可上線單位 |
|---|---|---|---|
| **F1 — 世界切換基礎** | `User.activeWorld`(nullable) + `TaskHistory.world` + 完成時寫入 + **World Picker（共用元件）** + backfill | 改 2 表 | 切換器能用 |
| **F2 — 公仔衍生（唯讀養成）** | `figureWorld.js`（階段/部位/進度 TDD）+ 小芽基底 SVG（6 階 + 9 配件）+ 自動成長渲染 + 餵養動畫 + 聚合 API | 零（衍生） | 萌寵隨完成數自動長大，先單一物種 |
| **F3 — 多物種 + 切換** | `User.activeFigure` + 奶茶 / 啵啵 基底 SVG + 物種解鎖（衍生）+ 切換 UI（re-skin 同池）+（可選）手動裝飾佈置 | 加 1 欄（+ 可選 1 表） | 可養 / 切多隻萌寵 |
| **F4 — 打磨 + 形態傾向** | 形態傾向（軸③）變體、進化過場動畫、季節限定、空狀態、bundle 預算 | 零 | polish |

**F1 ＝ 與居家世界 H1 同一塊**，動工前必須先與居家世界 session 對齊（§9.2）。F2 起公仔世界專屬，可獨立推進。

## 9. 開放問題（待 review / 跨 session 協調）

1. ~~**舊資料 / 選世界前歸屬**~~ **【沿用居家 §1.2 定案 — 共同序章】** `world IS NULL` ＝ 共享序章；公仔世界讀 `world='figure' OR world IS NULL`。
2. **World 切換基礎由誰實作（F1 ≡ H1）** — 居家 + 公仔都需要。建議**抽成獨立「world-switch 基礎」PR，兩方共依賴**（含 schema `activeWorld`/`world` + World Picker 共用元件）。需與居家世界 session 拍板誰先動 / 是否獨立 PR。**本 spec 與居家 spec §1 契約逐字一致，可直接當對齊基準。**
3. **形態傾向（軸③）規則** — 「主導 domain 決定」還是「平衡度決定」還是混合？v1 不做（列 F4/v2），先做部位疊加（軸②）。
4. **多物種：re-skin 同池 vs 每隻獨立成長池** — v1 傾向 re-skin 同池（零損失、零 schema、簡單）；獨立池較像收集但分散 + 需 per-pet schema。待 PM 定。
5. **公仔 ↔ 居家整合** — 未來萌寵是否站進居家房間？居家 spec §9.3 已留「角色層」。v1 不做，但兩邊 SVG 座標系 / 角色資產格式留擴充餘地。

## 10. 與既有系統的一致性檢查

- ✅ 重用 `categoryToDomain()`（Slice P）做 domain 正規化。
- ✅ 重用 `CATEGORY_CONFIG` 色票（tailwind content 已掃 lib）。
- ✅ 純函式 lib（`figureWorld.js`）+ 集中常數 + 確定性 + TDD — 對齊 `journeyWorld.js` / `homeWorld.js` 慣例。
- ✅ 側邊欄 `setCurrentView` 模式，不引入 bottom-tab。
- ✅ 零懲罰、opt-in、工具/收集分離 — 對齊 gamification note。
- ✅ World-switch 契約**與居家 spec §1 逐字一致** — 不分岔。
- ⚠️ 與居家世界**同樣**需改 `TaskHistory` + `User`（world attribution）——這是世界系統第一次為遊戲化改核心 schema，PM 已確認接受（換來可切換 + per-world 進度）。F1 兩世界共用這次 schema 變更。
